# Atlassian Team Playbook — DACI Decision-Making

## Source
- Author: Atlassian
- URL: https://www.atlassian.com/team-playbook/plays/daci
- Rights note: Metadata and original analysis only. Link to the source instead of copying it wholesale.

## Why this is good
DACI works as an operating document because it turns fuzzy collaboration into named roles. The writing is useful when teams confuse inclusion with decision clarity.

## Craft moves
- Defines roles around a decision, not hierarchy.
- Separates driver, approver, contributors, and informed parties.
- Makes collaboration explicit before conflict appears.
- Gives teams a reusable meeting/workshop shape.
- Reduces ambiguity in cross-functional work.

## Structure map
1. Name the decision.
2. Assign the driver.
3. Identify approver and contributors.
4. Clarify who only needs to be informed.
5. Document outcome and next steps.

## Excerpt or summary
Summary: Atlassian’s DACI playbook is a model for internal decision docs: roles, scope, and accountability before consensus theater takes over.

## Reusable pattern
For a decision doc: “Decision: [choice]. Driver: [owner]. Approver: [final authority]. Contributors: [inputs]. Informed: [stakeholders]. Due by: [date].”

## Imitation prompt
Write an internal DACI-style decision brief. Make the decision, roles, authority, and next action unambiguous.

## Anti-patterns to avoid
- Listing everyone as responsible.
- Confusing stakeholder input with approval rights.
- Leaving the actual decision unnamed.
