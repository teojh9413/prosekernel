# GitHub Changelog — Release Note Cadence

## Source
- Author: GitHub
- URL: https://github.blog/changelog/
- Rights note: Metadata and original analysis only. Link to the source instead of copying it wholesale.

## Why this is good
GitHub Changelog works because it treats each update as a compact promise: what changed, who it affects, and where to learn more. The cadence builds trust by making product change legible without turning every release into a launch essay.

## Craft moves
- Uses consistent update units so readers can skim by relevance.
- Puts the concrete product change before commentary.
- Links outward for detail instead of bloating the recurring feed.
- Maintains a factual voice suited to developers who need signal more than hype.
- Makes accumulated product progress visible through cadence.

## Structure map
1. Name the product or surface that changed.
2. State the change in one direct sentence.
3. Explain affected users or workflows.
4. Provide a link or next action for deeper context.
5. Return to cadence rather than over-selling the update.

## Excerpt or summary
Summary: GitHub’s changelog turns product updates into a repeatable communication rhythm. Its craft lesson is disciplined usefulness: each item earns attention by being specific, brief, and actionable.

## Reusable pattern
For a technical update newsletter: “Changed: [specific surface]. Impact: [who/what workflow]. Why it matters: [one practical implication]. Do next: [link/action].”

## Imitation prompt
Write a technical changelog newsletter item using GitHub’s cadence discipline. Lead with the concrete change, clarify who is affected, avoid launch hype, and give one useful next action.

## Anti-patterns to avoid
- Turning routine updates into inflated announcements.
- Omitting who the change matters for.
- Using vague “improved experience” language without the actual change.
