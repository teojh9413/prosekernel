# GOV.UK Design System — Error message

## Source
- Author: Government Digital Service
- URL: https://design-system.service.gov.uk/components/error-message/
- Rights note: GOV.UK Design System content is published for public reuse under GOV.UK terms; store link and analysis, not unnecessary copies.

## Why this is good
The copy treats an error as a repair moment. It does not shame the user, apologize for validation, or bury the fix. It tells the user exactly what is wrong and what input will satisfy the system.

## Craft moves
- Places the error next to the field where action is needed.
- Uses direct language that names the missing or invalid input.
- Avoids cute tone, blame, and unnecessary apology.
- Pairs summary-level orientation with field-level instruction.
- Makes the next action obvious enough to complete under stress.

## Structure map
1. User tries to submit or continue.
2. System identifies the exact field with a problem.
3. Copy states the required correction in plain language.
4. Interface returns focus to the place where the user can fix it.
5. User completes the task without guessing.

## Excerpt or summary
Summary: effective error messages are specific, located near the problem, and written as instructions for recovery rather than as generic failure notices.

## Reusable pattern
“[Field/problem]: Enter/select/provide [specific required thing].” If the user can recover, the copy should point directly to the recovery action.

## Imitation prompt
Rewrite these product errors in GOV.UK style: name the field, state the issue, give the exact recovery action, remove jokes, blame, and vague “something went wrong” language.

## Anti-patterns to avoid
- “Oops!” or playful error copy in serious task flows.
- Telling users something is invalid without saying how to fix it.
- Putting the error far away from the field that needs action.
