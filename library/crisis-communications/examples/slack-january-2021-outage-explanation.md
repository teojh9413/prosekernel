## Why this is good
Slack’s outage explanation is valuable because it balances accessibility and technical depth. It gives enough orientation for affected users while preserving the engineering specifics needed for credibility.

## Craft moves
- Acknowledges user impact before technical detail.
- Explains sequence in reader-friendly layers.
- Uses technical specificity as credibility.
- Connects fixes to the cause.

## Structure map
1. What users experienced.
2. What happened internally.
3. Why it happened.
4. What was fixed.
5. What changes next.

## Excerpt or summary
Incident explanation analysis only; no source text mirrored.

## Reusable pattern
Layer incident communication from user-visible impact to technical cause to concrete prevention.

## Imitation prompt
Write an outage explanation that starts with user impact, then layers technical cause, fixes, and future prevention.

## Anti-patterns to avoid
- Starting with infrastructure jargon before user impact.
- Oversimplifying until credibility disappears.
- Ending without concrete changes.
