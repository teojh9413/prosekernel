## Why this is good
The update works because it combines candor with actionability. Users need to know what changed, whether they are affected, what action to take, and how to verify safety. Security communication fails when it either hides risk or creates confusion.

## Craft moves
- States the triggering event and corrective action.
- Gives users concrete verification steps.
- Separates known facts from broader security implication.
- Uses calm precision suited to security-sensitive workflows.
- Links to support details rather than burying instructions.

## Structure map
1. State what changed.
2. Explain why it changed.
3. Clarify who is affected.
4. Give exact user actions or verification steps.
5. Point to ongoing updates/support.

## Excerpt or summary
Summary: GitHub’s SSH host key update is useful as crisis microcopy: it turns an alarming security-adjacent event into precise user instructions.

## Reusable pattern
For security customer action: “We changed [asset] because [reason]. If you use [workflow], you may see [signal]. Verify by [steps]. If [condition], do [action].”

## Imitation prompt
Write a security update for users in GitHub’s calm-action style. Preserve facts, give verification steps, and avoid vague reassurance.

## Anti-patterns to avoid
- Creating fear without instructions.
- Assuming all users understand the technical signal.
- Hiding required action below corporate apology text.
